    Why I Built It — The Density of Judgment

    Problem

    Ten years of running construction sites taught me that a good engineer is not the one who builds fastest, but the one who judges correctly. When AI ships code in seconds, a junior's survival strategy hinges on how densely and quickly they can layer judgment on top of generated code.

    Approach

    I systematized the judgment loop itself for a team of five juniors with no senior. A strict gate that only allows commits at ≥ 80 / 100 raised everyone's standard. Not a code dump — an open-sourcing of the judgment logic itself.

    Core Design Principle — "Only teach Claude what it doesn't know"

    Running Anthropic's skill-creator on the plugin revealed that 55.6% of the original 54 tools were "Encoded Preferences" — things Claude already knows how to do but that we were forcing into a team-specific way. The redesign rule: facts Claude can't infer → keep as Skill, things Claude can do but we want a specific way → convert to Agent, things Claude already does well → drop. 22 Skills removed; 7 kept.

    Using Claude Code's trigger mechanisms

    • Skills — Claude autonomously decides to load (only things Claude cannot infer)
    • Agents — description always exposed; injected on match (enforces the team's way)
    • Commands — invoked explicitly by the user (/prd, /implement, /auto-commit)

    Core Pipeline — /prd → /implement → /auto-commit

    /prd — Planning Automation

    Converts a natural-language idea into a structured PRD. /digging analyzes the PRD's weaknesses across four categories (completeness, feasibility, security, consistency) and blocks problems before they reach implementation.

    /implement — Team Parallel Build

    Parses the PRD, dynamically assigns Backend / Frontend / AI / Ops teams, and runs them in parallel. SHARED_CONTEXT exchanges API contracts in real time so the frontend can reference an API the backend has just produced.

    /auto-commit — Quality Gate

    A 100-point code review. Commits require ≥ 80; 60 – 79 triggers an auto-improvement pass; below 60 is blocked. Any Security Critical issue forces a block regardless of the score.

    3-Tier Memory System

    Structure

    • Layer 1 — Auto Memory (MEMORY.md): persistent project knowledge across sessions (core stack, repo patterns)
    • Layer 2 — SHARED_CONTEXT: real-time inter-agent sharing (e.g., backend API contracts referenced live by the frontend)
    • Layer 3 — TaskCreate: tracks task dependencies inside the current conversation; auto-runs the next task when a prerequisite completes

    Effect

    The result is a "living development team" that understands the project better the more it runs. Architecture decisions that used to take days now resolve in five minutes of validation. End-to-end pipeline speed improved by 3×.
