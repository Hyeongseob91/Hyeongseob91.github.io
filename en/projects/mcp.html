VALORITHM — MCP-based AI System for Game Development

Wanted Learning Project Lead / Tech Lead (AI) Apr – Jun 2025 MCP Server + LangGraph Agent
MCP LangGraph RAG FastMCP NumPy ChromaDB Whisper Unreal Engine FastAPI

Contents

    Problem

    The biggest bottleneck in our FPS-game development was repetitive busywork.
    Setting the recoil pattern of a single weapon took ~40 minutes; if a designer asked to "make it kick higher," the engineer had to retune values and retest in a loop.
    3D-map whiteboxing — sightline checks, prop placement, custom tweaks — took ~8 hours per map iteration.

    So we set out to automate this repetitive work with AI. Free up engineering bandwidth, generate opportunity cost for the company, and let engineers focus on the work that matters.

    Solution

    What we built

    • Three AI tools that support planning + development for shipping a new game
    • Natural-language interfaces via MCP (Model Context Protocol) tool integration
    • An in-game AI agent "Javis" built on LangGraph + STT

    Core value

    Integrate MCP tools and AI agents across the project to make planning-through-play workflows intelligent — a new game-development paradigm where ideas can be implemented immediately, without the friction of repetitive work.

    Highlights

    • Weapon-recoil generation — 40 min → 30 s (~98.7% faster)
    • 3D-map whiteboxing — 8 h → 2 h (~75% faster)
    • MCP server release — published on the Smithery.ai marketplace
    • 3 AI tools integrated — Discord Agent + Recoil Generator + 2D→3D Map Generator
    • MCP-standard adoption — compatible with IDEs and external LLM environments

    AI Tools Overview

    1. Discord MCP Agent

    • Auto-summarize meeting notes and send schedule reminders
    • Claude API + Discord MCP Server integration
    • Dual storage: Oracle RDB + ChromaDB

    2. Weapon Recoil Generator

    • Natural-language commands generate per-weapon recoil trajectories
    • NumPy-based 3-stage burst pattern (early/middle/late shots)
    • Matplotlib visualization → applied directly in Unreal Engine

    3. 2D-to-3D Map Generator

    • Auto-generate a 3D mesh (.obj) from a single 2D image
    • OpenCV Canny edges + Shapely / Open3D

    4. Javis AI Agent — PoC

    • In-game AI agent "Javis" built on LangGraph + STT
    • PoC only within our timeline; not applied in the actual game

    What I Built & Technical Deep Dive

    MCP-based AI system design

    • Built the MCP (Model Context Protocol) server connecting Claude and Unreal Engine
    • Server-side tool registration and schema automation via the FastMCP library
    • Direct engine integration through the open-source "Unreal MCP Plugin"

    Weapon Recoil Generator

    • NumPy-based 3-stage (early/middle/late) recoil-weight algorithm
    • Continuous recoil-trajectory coordinates via np.cumsum()
    • - Early: minimal X-axis sway; vertical Y-axis recoil concentration
    • - Middle: stabilization zone, uniform distribution
    • - Late: stronger X-axis sway, harder to control
    • Workflow that pushes assets directly into the engine via the Unreal MCP Plugin

    LangGraph Agent (PoC)

    • TypedDict-based agent state and multi-turn conversation logic

    Lessons learned

    I didn't yet know how to integrate with Unreal Engine when I first designed the MCP tooling. We discovered the open-source "Unreal MCP Plugin" and designed around invoking remote tools from inside the engine.
    Trouble started there: the plugin was a personal project by a US college student and not really a finished product. After several attempts we still could not get reliable Remote communication between Unreal and the MCP server.
    We pivoted to local communication and revised the plan.

    The lesson: when leaning on open source, do thorough up-front validation. To respond flexibly to unexpected situations, you need solid foundational engineering — that is what lets you propose alternatives.

    Technical growth

    Most teams just call MCP tools others have built. Designing and operating an MCP server from scratch deepened my understanding of the protocol and its internals.
    The biggest personal gain was understanding what a "server" actually is. Smithery.ai's marketplace, where I deployed, provides its own server to attract developers — I didn't appreciate the value during build, only after release.

    That gave me a maintenance-and-operations lens on servers and a clearer sense of what scalable design looks like through the server abstraction. Going forward, I want to bring Infrastructure-as-Code thinking into system design from day one.
