    Overview

    An entry for Google's global Gemini Live Agent Challenge.

    Architecture — Dual pipeline

    Not a single path (Live API only) — a multi-path design built for flaky museum Wi-Fi.

    Gemini Live API pipeline (main)

    • Voice input (PCM16/16 kHz) and camera (JPEG @ 1 fps) processed in real time over the same WebSocket session
    • 4 Function Calls replace orchestration — routing logic, intent classifier, and state machine are all gone
    • Barge-in lets the user cut off the AI response and ask the next question naturally

    ADK multi-agent pipeline (text fallback)

    • timelens_orchestrator → 4 sub-agents (curator / restoration / discovery / diary)
    • Shares the same backend APIs as the Live path — no code duplication
    • Whichever entry point runs, the same functions are called — only the entry differs

    Function Calling Workflow — 4 tools

    The original plan included an intent classifier, an if/else chain, and a state machine. In the end we left only the system prompt and 4 Function Declarations and let the model handle the rest — and it worked better. The takeaway: "the model is smarter than your routing logic."

    The 4 tools

    • recognize_artifact() — analyze the camera frame + Google Search Grounding for artifact recognition and lookup
    • generate_restoration() — generate a photorealistic restored image with Gemini 2.5 Flash Image
    • discover_nearby() — GPS-driven Google Places API lookup of nearby museums and heritage sites
    • create_diary() — at the end of the session, auto-generate a visit diary with Gemini 3 Pro Image

    Agentic Camera

    Voice triggers the shutter — the central UX. Per-language regex (Korean: "이거 뭐야", "이거 봐" / English: "what is this", "look at this" / Japanese, Chinese, Hindi, etc.) detects voice commands across 15 languages.

    Capture flow

    • Voice command detected → 500 ms stabilization wait (suppresses hand shake)
    • High-resolution capture → iOS-style white flash overlay (0.2 s fade)
    • Frame plus the user's utterance sent to the Live API → AI voice response

    Service Flow

    STEP 1

    Landing — Pick a language

    Five languages: Korean / English / Japanese / Chinese / Hindi. "Museums Come Alive" — request camera and microphone permissions, start the session.

    STEP 2

    Permissions

    Grant camera and microphone access. If denied, automatically falls back to text mode (ADK).

    STEP 3

    Service overview

    "Point your camera at an artifact, and a thousand-year story begins." Average 3-second recognition, 10K+ heritage sites, 24/7 AI voice docent.

    STEP 4

    Session init

    Connect the Gemini Live session. Detect the current museum from GPS and wake up the AI curator.

    STEP 5

    Curator greeting

    The current exhibition info is fetched live via Google Search Grounding and rolled into the greeting. Two-way voice and text are both available.

    STEP 6

    Artifact recognition

    "What is this?" — the agentic camera detects the voice command, captures, and calls recognize_artifact(). The curator explains the artifact's name, era, and historical context.

    STEP 7

    Real-time multimodal conversation

    In LIVE mode, camera frames and voice are processed concurrently. Look at an artifact, ask freely, and barge in to start a new question.

    STEP 8

    Restored image

    "Show me what it originally looked like" — generate_restoration() produces a photorealistic restored image with Gemini 2.5 Flash Image. Toggle Now / Original to compare.

    Problems we solved along the way

    Korean STT transcription quality

    Gemini outputTranscription mishandles Korean ("괜찮 ." → "괜찮.", "큐레 이터" → "큐레이터"). Fixed with post-processing — strip spaces before punctuation, merge wrongly-split syllables — plus a system-prompt rule capping sentences to ≤ 15 words.

    Museum Wi-Fi

    Firebase createSession() hangs indefinitely on slow connections. Fixed with a Promise.race 5-second timeout → fall back to a local session. The user never notices.

    ADK vs Zod version conflict

    ADK's internal zod/v3 conflicted with our Zod v4. Resolved by using @google/genai's Schema interface directly.
