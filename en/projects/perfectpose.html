Contents

    Problem

    Project background

    "Plan to deploy in 24 hours." A Fast Builder Challenge mission.
    We had to ship an end-to-end service in 24 hours, surveyed many references, and chose to build a rhythm game inspired by Steam's "Perfect Poses" — players match the on-screen pose, AI scores accuracy in real time.

    Technical challenge

    The hardest part was the real-time data bridge between the AI team (Python) and the Unreal team (C++).
    Capture webcam frames at 30 fps, detect with YOLO-Pose, and ship 17 keypoints over an optimized WebSocket to Unreal Engine without lag — all to deliver a polished gameplay experience inside a tight window.

    Fast Builder mindset

    We prioritized a working prototype over perfect code.
    The core was clear interface definitions for fast collaboration decisions under time pressure.

    Solution

    What we built

    • YOLO11-Pose-based real-time engine that detects 17 joint coordinates
    • FastAPI async WebSocket server (AI ↔ Unreal bridge)
    • Player segmentation with Meta AI's SAM
    • Unreal Engine 5 game client (pose matching + scoring)
    • Pose-feedback generation with Bllossom (Korean LLM)

    Core value

    Inside 24 hours we completed the entire end-to-end pipeline — plan → design → develop → test → deploy.
    To enable the AI and Unreal teams to work in parallel, we defined the API interface first and integrated independently developed pieces afterward — a small simulation of cross-team collaboration in real production.

    Highlights

    • Build window — E2E game service shipped in 24 hours (plan → deploy)
    • API response time — average < 50 ms
    • Real-time processing — 30 fps webcam-based YOLO-Pose detection
    • Stack integration — YOLO-Pose + SAM + FastAPI + Unreal Engine 5
    • Team coordination — 3 AI + 3 Unreal in parallel

    Project Workflow

    Data flow

    Webcam (30 fps) → OpenCV preprocessing → YOLO11-Pose inference → 17 keypoints → JSON serialization → FastAPI REST API → Unreal Engine rendering → pose match & score

    Communication protocol

    The AI server and Unreal client communicate over JSON over HTTP. Unreal periodically polls /api/pose for the latest pose data. Real-time game requires minimal response delay, so async processing and a singleton model-instance pattern were applied.

    What I Built

    1. Project Leading (PL)

    Leveraging prior PM experience, I led the project end-to-end.

    • Kickoff & scope: reference analysis → AI core feature alignment → MVP scope
    • Team & roles: 3 AI / 3 Unreal — clear responsibilities, parallelizable tasks
    • Interface decision: finalized AI ↔ Unreal API spec within 3 hours so both teams could work independently
    • Schedule: split 24 hours into 4 phases (plan / dev / integration / wrap-up); presentation strategy per phase
    • Risk: integration test scheduled mid-stream so issues surfaced early

    2. YOLO-Pose Engine

    Real-time pose-detection engine in the PoseEstimator class.

    • YOLO11n-pose (lightweight enough for real-time)
    • 17 COCO Keypoints: nose, eyes, ears, shoulders, elbows, wrists, hips, knees, ankles
    • Filter by confidence (conf_threshold=0.5)
    • Singleton pattern to avoid model-load overhead
    • OpenCV-driven frame processing and keypoint visualization

    Lessons learned

    The original plan had Unreal driving a character that mimics the player from YOLO-Pose keypoints. Integration testing exposed the issue: AI-server keypoint deltas were correctly received, but the Unreal character UI couldn't translate them into natural motion. Realistic character motion needs a deep understanding of joint rotations and bone structures — not feasible in 24 hours.
    That was my first real moment of panic. With zero game-development experience, I didn't know what alternative to propose.

    Quick teamwork pulled us through: we simplified the UI to point-to-point laser links instead of full character animation. The result felt low-fi for an Unreal Engine project, but the core game logic and pose recognition shipped on time. The takeaway: at MVP stage, focus on the core function before chasing polish.

    Technical growth

    I learned firsthand that a working prototype can matter more than perfect code.
    What you choose to build — and what you choose to drop — determines whether a project completes inside a tight window.

    Specifically, defining the interface first turned out to be critical for collaboration: with the API spec locked in early, the AI and Unreal teams could work independently without waiting on each other.

    The other learning was the integration test. Each part worked alone; problems only surfaced once we wired them together. If we'd put testing at the very end, the project would not have shipped. Since then, TDD and unit/integration testing make sense to me as a foundation for higher-quality code.
