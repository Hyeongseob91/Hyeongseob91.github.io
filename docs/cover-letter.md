# 현장의 문제 해결 경험을 AI 시스템 설계로 연결하는 엔지니어, 김형섭입니다

## 1. 개인적 성찰과 협업 역량

### [Q1] 개인적 성장 경험: 건축 현장에서 AI 엔지니어로의 전환

#### 변화의 시작점이 된 결정적 계기

8년간 건설업에서 현장 엔지니어와 프로젝트 매니저로 일하며, 저는 항상 "반복되는 문제를 어떻게 시스템으로 해결할 수 있을까"를 고민했습니다. 현대건설에서 기술연구소의 층간 차음 신기술 'H-Silence' 현장 실증을 리드하며 국내 최초 국토교통부 1등급 인증을 획득했을 때, 기술이 현장의 문제를 근본적으로 해결할 수 있다는 확신을 얻었습니다.

그러던 중 LLM과 Agent 기술을 접하게 되었고, 제가 현장에서 해왔던 "문제 인식 → 판단 → 조율"의 과정이 AI 시스템의 "추론 흐름과 의사결정 구조"와 본질적으로 같다는 것을 깨달았습니다. 이것이 커리어 전환의 결정적 계기가 되었습니다.

#### 과정 속의 고뇌와 감정적 성숙

30대 중반에 완전히 새로운 분야로 전환한다는 것은 쉬운 결정이 아니었습니다. 8년간 쌓아온 경력을 뒤로하고, 다시 처음부터 시작해야 한다는 두려움이 있었습니다. 하지만 저는 "기초를 가장 중요하게 생각하는 사람"입니다. 어떤 기술이든 근본적인 질문에 대한 이해 없이 쌓아 올려진 구조는 반드시 한계에 부딪힌다고 믿기 때문입니다.

Wanted Learning K-Digital Training 과정을 거치며 AI/ML의 기초부터 다시 쌓았고, 빅데이터 분석기사 자격증을 취득하며 이론적 토대를 마련했습니다.

#### 현재의 나에게 미친 지속적인 영향력

현재 저는 (주)사운드마인드에서 AI Research Engineer로서 High-Performance RAG Engine 아키텍처를 설계하고, Enterprise RAG Agent PlayGround를 Project Lead로 이끌고 있습니다. 건설 현장에서 익힌 "복잡한 이해관계 조율"과 "프로젝트 전체 흐름 관리" 역량이 AI 시스템 설계에 그대로 적용되고 있습니다.

### [Q2] 인간관계 경험: WIGTN 크루를 통한 주니어 개발자 성장 생태계 구축

#### 협업 과정에서 마주한 잊지 못할 순간

저는 **WIGTN**이라는 주니어 개발자 크루를 이끌고 있습니다. AI 시대에 주니어 개발자가 나아가야 할 방향은 AI를 활용하는 역량에 달려 있다고 생각했기 때문입니다. 크루를 운영하며 가장 인상 깊었던 순간은, 처음에는 "AI가 개발자를 대체하는 것 아니냐"며 불안해하던 멤버들이 Claude Code Skills Plugins을 직접 사용해보고 "AI는 위협이 아니라 강력한 파트너"라는 것을 체감했을 때입니다.

#### 상황에 직면했을 때의 구체적 대응 기제

주니어 개발자들이 AI 도구를 효과적으로 활용할 수 있도록, 저는 **Claude Code Skills Plugins** (wigtn-plugins-with-claude-code)를 직접 개발했습니다. 이 플러그인은 체계적인 개발 워크플로우를 자동화합니다:
- `/prd`: 요구사항 문서 자동 생성
- `/digging`: PRD 취약점 분석
- `/implement`: 구현 계획 수립
- `/auto-commit`: 품질 검증 후 자동 커밋

#### 관계를 통해 체득한 협업의 원칙과 교훈

크루를 운영하며 깨달은 것은, "시니어가 되기까지 10년이 걸렸던 경험의 축적을, AI라는 강력한 파트너와 함께한다면 5년, 3년으로 압축할 수 있다"는 것입니다. 저와 같은 고민을 하는 주니어, 비전공자 분들에게 작은 도움이 될 수 있다면 좋겠다는 마음으로 오픈소스 활동을 지속하고 있습니다.

### [Q3] 의미 있는 프로젝트: Enterprise RAG Agent PlayGround

#### 활동의 시작 동기와 목표 설정

사운드마인드에서 AI Research Lead로서 마주한 첫 번째 과제는 "AI 기술을 어떻게 고객에게 증명할 것인가"였습니다. AI 기술은 추상적이어서 고객사에게 설명하기 어렵고, 제안서만으로는 기술력을 증명하기 힘들었습니다. 또한 기본적인 RAG 시스템은 검색 정확도가 낮고, LLM이 생성한 답변의 근거를 검증할 방법이 없었습니다.

#### 목표 달성을 위해 거쳐온 핵심적인 수행 과정

저는 "왜 이 답이 나왔는지 검증 가능한 UX"를 핵심 가치로 설정하고, End-to-End RAG 시스템을 Product-grade로 구현했습니다:

- **Semantic Chunking + Safety Guard**: 의미 단절점 기반 문서 분할, 2-Stage Chunking 구조
- **Advanced Hybrid Retrieval**: LLM 기반 Multi-Query Rewrite, Dense+Sparse+RRF 결합, Cross-encoder Reranking
- **Observability & Trust UX**: 파이프라인 단계별 Latency 모니터링, Retrieval Insight 시각화

5계층 MSA 아키텍처(Presentation → API Gateway → Service → Data → External Model Services)로 설계하여, 확장성과 유지보수성을 확보했습니다.

#### 결과가 가져온 변화와 개인적 보람

현재 이 시스템은 B2B 영업 도구, 요구사항 수집 도구, 사내 Ideation 도구로 활용되고 있습니다. "AI 기술의 비즈니스 가치를 증명하는 도구"로서, 회사의 기술경쟁력 제고에 기여하고 있습니다. 2026년 상반기 AI Platform R&D 전략 및 로드맵 수립을 리드하며, 기술과 비즈니스를 연결하는 역할을 수행하고 있습니다.

## 2. [Case Study] 문제 해결 프로세스: Simple LLM Loadtester 개발기

> **Note:** 이 섹션은 Claude Code Skills Plugins을 활용하여 2일 만에 완성한 오픈소스 프로젝트의 문제 해결 과정을 담고 있습니다.

### [Q4] 문제의 발견: 기존 LLM 벤치마킹 도구의 한계

#### 사용자 입장에서 느낀 불편함과 문제 상황의 재구성

금요일 퇴근 무렵, 상사분의 질문에서 이 프로젝트가 시작되었습니다: **"우리 LLM 서버, 동시 접속자 몇 명까지 가능할까요?"**

규모가 작은 조직에서는 "인프라 평가 / 성능 테스트"가 담당자에게 축적되기보다, 필요할 때마다 누군가가 임시로 맡아서 처리하게 됩니다. 기존 벤치마킹 도구들을 조사해보니 세 가지 문제가 있었습니다.

#### UI/기능/데이터 구조 측면의 복합적 원인 진단

| 문제 영역 | 상세 내용 |
|----------|----------|
| **기술적 문제** | 기존 도구들은 개발자 친화적이라, 비개발직군의 LLM 특화 메트릭(TTFT, TPOT, ITL) 시각화 지원이 부족 |
| **접근성 문제** | CLI 기반에 복잡한 설정이 필요해, 소규모 연구원이나 비전공자가 쉽게 접근하기 어려움 |
| **품질 평가 문제** | 단순 처리량만 측정하고, SLO 기반 품질 평가(Goodput)를 지원하지 않음 |

#### 이상적인 시스템 작동 방식에 대한 정의

**"비개발자도 누구나 돌려볼 수 있는 사내 공용 LLM LoadTester"**를 만들어보자는 생각으로 주말 프로젝트를 시작했습니다. 타겟 사용자를 명확히 정의했습니다:
- **주니어 개발자**: AI 협업 개발 방법론 학습 및 실습
- **소규모 연구원**: 복잡한 설정 없이 LLM 서빙 성능 실험
- **비전공자/기획자**: 직관적인 UI로 LLM 서비스 도입 검증

### [Q5] 전략적 접근: AI-Native Fast Build 방법론

#### 핵심 기능 우선순위 선정 기준과 판단 근거

WIGTN 크루에서 개발한 Claude Code Skills Plugins을 활용하여, 체계적인 개발 워크플로우를 적용했습니다:

1. **Day 1**: `/prd`로 요구사항 정의 → `/digging`으로 PRD 취약점 분석 → 아키텍처 설계 → 백엔드 핵심 로직 구현
2. **Day 2**: 프론트엔드 대시보드 → 통합 테스트 → Docker 배포 설정 → `/auto-commit`으로 품질 검증

#### 최소 기능 제품(MVP) 구현 및 초기 배포 전략

MSA 아키텍처(API Service + Web Service)로 설계하고, 다음 핵심 기능을 구현했습니다:
- OpenAI 호환 API 서버 지원 (vLLM, SGLang, Ollama, LMDeploy, TensorRT-LLM)
- LLM 특화 메트릭: TTFT, TPOT, E2E Latency, ITL, Throughput
- Goodput 메트릭: SLO 임계값 기반 품질 평가
- 실시간 WebSocket 기반 진행 상황 모니터링
- GPU 메트릭 수집 (메모리, 사용률, 온도, 전력)

#### 전체 로드맵 중 나머지 기능들의 단계별 확장 계획

향후 계획으로 Triton Inference Server 어댑터 완성, 벤치마크 비교 기능 추가, 커뮤니티 피드백 반영을 설정했습니다.

### [Q6] 가설 검증: AI 협업 개발의 효과성 검증

#### 초기 가설 수립

**"Claude Code Skills Plugins을 활용하면, 주니어 개발자도 2일 만에 완성도 높은 풀스택 애플리케이션을 구축할 수 있다."**

#### 실제 데이터 결과와의 간극 및 가설 재검토 과정

가설은 검증되었습니다. AI 협업 개발은 단순히 코드를 대신 작성하는 것이 아니었습니다. PRD 작성, 아키텍처 검토, 코드 리뷰, 테스트 등 전체 개발 라이프사이클을 가속화했습니다.

특히 `/digging` 명령으로 PRD 취약점을 사전에 발견하고, `/auto-commit`의 품질 검증으로 코드 품질을 유지할 수 있었습니다.

#### 다각도 분석을 통한 인사이트 도출

| 기술 스택 | 선택 이유 |
|----------|----------|
| **FastAPI** | 비동기 설계(asyncio + httpx)로 고동시성 부하 생성 |
| **Next.js** | Recharts를 활용한 인터랙티브 차트 시각화 |
| **WebSocket** | 벤치마크 진행 상황 실시간 전달 |
| **어댑터 패턴** | 다양한 서버 타입 지원을 위한 확장 가능한 구조 |
